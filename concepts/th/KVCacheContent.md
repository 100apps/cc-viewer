# เนื้อหา KV-Cache

## KV-Cache คืออะไร?

KV-Cache (Key-Value Cache) เป็นกลไกหลักสำหรับการเร่งความเร็วระหว่างการอนุมานในโมเดลภาษาขนาดใหญ่ เมื่อคุณสนทนากับ Claude ทุกคำขอจะส่งประว��ติการสนทนาทั้งหมด (system prompt + คำจำกัดความเครื่องมือ + ข้อความในอดีต) หากต้องคำนวณทุกอย่างใหม่ทุกครั้ง มันจะใช้เวลานานและมีค่าใช้จ่ายสูง

บทบาทของ KV-Cache คือการเก็บเนื้อหาที่คำนวณแล้วไว้ในแคช เพื่อให้สามารถนำกลับมาใช้ใหม่ในคำขอครั้งต่อไป โดยข้ามการคำนวณซ้ำ

## วิธีการทำงานของแคช

Prompt caching ของ Anthropic รวมคีย์แคชในลำดับคงที่:

```
System Prompt → Tools → Messages (จนถึงจุดแบ่งแคช)
```

ตราบใดที่คำนำหน้านี้เหมือนกับคำขอก่อนหน้า API จะโจมตีแคช (ส่งกลับ `cache_read_input_tokens`) ซึ่งลดความล่าช้าและค่าใช้จ่ายอย่างมาก

## "เนื้อหา KV-Cache ปัจจุบัน" คืออะไร?

"เนื้อหา KV-Cache ปัจจุบัน" ที่แสดงใน cc-viewer คือเนื้อหาที่แยกออกมาจากคำขอ MainAgent ล่าสุดและทำเครื่องหมายว่าเป็นเนื้อหาที่แคช โดยเฉพาะรวมถึง:

- **System Prompt**: คำแนะนำระบบของ Claude Code รวมถึง CLAUDE.md ของคุณ คำอธิบายโครงการ ข้อมูลสภาพแวดล้อม เป็นต้น
- **Messages**: ส่วนที่แคชของประวัติการสนทนา (โดยปกติคือข้อความที่เก่ากว่า)
- **Tools**: รายการคำจำกัดความเครื่องมือที่พร้อมใช้งาน (เช่น Read, Write, Bash, เครื่องมือ MCP เป็นต้น)

เนื้อหานี้ถูกทำเครื่องหมายในเนื้อหาของคำขอ API ผ่าน `cache_control: { type: "ephemeral" }` เพื่อระบุขอบเขตแคช

## ทำไมต้องดูเนื้อหาแคช?

1. **ทำความเข้าใจบริบท**: รู้ว่า Claude "จำ" เนื้อหาใดในปัจจุบัน และช่วยให้คุณประเมินว่าพฤติกรรมนั้นตรงตามที่คาดไว้หรือไม่
2. **การปรับให้เหมาะสมของต้นทุน**: การโจมตีแคชมีค่าใช้จ่ายน้อยกว่าการคำนวณใหม่มาก การดูเนื้อหาแคชช่วยให้คุณเข้าใจว่าทำไมคำขอบางรายการจึงเรียกใช้การสร้างแคชใหม่
3. **การแก้ไขข้อบกพร่องในการสนทนา**: เมื่อคำตอบของ Claude ไม่ตรงตามที่คาดไว้ คุณสามารถตรวจสอบเนื้อหาแคชเพื่อยืนยันว่า system prompt และข้อความในอดีตถูกต้อง
4. **การนำทางคำแนะนำของผู้ใช้**: ผ่านรายการข้อความของผู้ใช้ในเนื้อหาแคช คุณสามารถข้ามไปยังตำแหน่งใดก็ได้ในการสนทนา
5. **การตรวจสอบคุณภาพบริบท**: ในระหว่างการแก้ไขข้อบกพร่องในแต่ละวัน การเปลี่ยนแปลงการกำหนดค่า Claude Code หรือการปรับ prompt KV-Cache-Text จะให้มุมมองที่เป็นศูนย์กลาง ซึ่งช่วยให้คุณยืนยันอย่างรวดเร็วว่าบริบทหลักไม่ได้เสื่อมลงหรือถูกปนเปื้อนด้วยเนื้อหาที่ไม่คาดคิด—โดยไม่ต้องตรวจสอบข้อความต้นฉบับทีละรายการ

## วงจรชีวิตของแคช

- **การสร้าง**: ในคำขอแรกหรือหลังจากความล้มเหลวของแคช API จะสร้างแคชใหม่ (`cache_creation_input_tokens`)
- **การโจมตี**: เมื่อคำขอต่อมามีคำนำหน้าที่สอดคล้องกัน แคชจะถูกนำกลับมาใช้ใหม่ (`cache_read_input_tokens`)
- **การหมดอายุ**: แคชมี TTL 5 นาที (ระยะเวลาการใช้งาน) และหมดอายุโดยอัตโนมัติหลังจากหมดเวลา
- **การสร้างใหม่**: เมื่อ system prompt รายการเครื่องมือ โมเดล หรือข้อความเปลี่ยนแปลง คีย์แคชไม่ตรงกัน ซึ่งเรียกใช้การสร้างใหม่
